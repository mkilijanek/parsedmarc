1. ARCHITEKTURA I STAN OBECNY
================================================================================

A. STRUKTURA PROJEKTU
---------------------
- Język: Python 3.12
- Framework: Flask 3.1.3 + SQLAlchemy 2.0.36
- Baza danych: PostgreSQL 16 z JSONB + pg_trgm
- Cache: Redis 7 (AOF, 512MB, LRU)
- Web Server: Nginx z TLS 1.2+/HTTP/2
- Task Queue: Wbudowany scheduler (schedule library)

2. ZNALEZIONE PROBLEMY I RYZYKA
================================================================================

A. PROBLEMY KRYTYCZNE (CRITICAL)
--------------------------------
1. CVE-2026-39892: Cryptography buffer overflow (requirements.txt ma 46.0.6,
   ale dependabot zgłasza podatność - WYMAGA AKTUALIZACJI)

2. CVE-2025-71176: pytest tmpdir handling (dev dependency, ale warto śledzić)

3. GHSA-v92g-xgxw-vvmm: Mako path traversal (przez alembic/dependency chain)

4. Manual sync clobbers feed env (#130) - dane konfiguracyjne mogą zostać
   nadpisane pustymi wartościami przy braku DB overrides

B. PROBLEMY WYSOKIEJ PRIORYTETOWOŚCI (HIGH)
-----------------------------------------
1. Nginx 502 po recreacji kontenera (#132) - wpływa na dostępność produkcyjną

2. Manual abuse.ch sync divergence (#131) - niespójność między workerem
   a ręcznym sync

3. Brak rozdzielenia logów feedów (#134) - trudność w diagnostyce

4. Dangerous wipe bez potwierdzenia tokena (#133) - ryzyko przypadkowego
   usunięcia danych

C. TECHNICZNY DŁUG (MEDIUM/LOW PRIORITY)
----------------------------------------
1. Dualizm schematu SQL/ORM - częściowo rozbieżność między database/init/
   a Alembic migrations

2. app/main.py wciąż zawiera ~2000 linii - można dalej uprościć

3. Wbudowany scheduler (schedule library) - brak rozdzielenia worker pool
   i brak retry dla failed jobs z backoff

4. Brak wersjonowania API - endpointy nie mają /v1/ prefixów

5. Config jako jedna klasa 100+ pól - trudny w utrzymaniu

6. Testy pokrywają 487 przypadków, ale brak testów integracyjnych z
   prawdziwym PostgreSQL

D. OBSERWACJE ARCHITEKTONICZNE
------------------------------
1. DOBRE:

   - Wyodrębnienie service layer (feed_ops, common)

2. WYMAGA POPRAWY:
