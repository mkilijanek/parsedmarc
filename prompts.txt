PROMPT 3.1: Diagnostyka Feed Sync Issues
----------------------------------------
Gdy feed zgłasza fetched=0 lub failure:

1. Sprawdź kroki:
   ```
   a. Czy feed jest enabled w DB?
   b. Czy env variables są ustawione (docker inspect)?
   c. Czy DB overrides nie nadpisują env pustymi wartościami?
   d. Czy connector działa bezpośrednio (test w kontenerze)?
   e. Czy job queue nie ma backlogu?
   f. Czy logs zawierają stop_reason?
   ```

2. Query dla diagnostyki:
   ```sql
   SELECT source, status, fetched_count, stop_reason, details
   FROM feed_runs ORDER BY finished_at DESC LIMIT 10;

   SELECT * FROM sync_jobs WHERE status = 'failed';

   SELECT * FROM app_logs WHERE level = 'ERROR'
   AND message ILIKE '%feed%' ORDER BY created_at DESC;
   ```

3. Kod do testowania bezpośredniego:
   ```python
   docker compose exec -T worker python - <<'PY'
   from app.services.abusech import update_abusech_indicators
   from app.db import SessionLocal
   db = SessionLocal()
   result = update_abusech_indicators(db)
   print(result)
   PY
   ```

---

PROMPT 3.2: Diagnostyka Nginx 502
---------------------------------
Gdy nginx zwraca 502 Bad Gateway:

1. Checklist:
   ```
   a. Czy app kontener działa? (docker compose ps)
   b. Czy app słucha na poprawnym porcie? (ss -lntp)
   c. Czy healthcheck przechodzi? (curl /healthz)
   d. Czy nginx ma poprawny upstream? (docker exec nginx cat /etc/nginx/conf.d/*)
   e. Czy IP kontenera się nie zmieniło? (docker network inspect)
   f. Czy SELinux nie blokuje? (ausearch -m AVC)
   ```

2. Fix - recreate nginx:
   ```bash
   docker compose stop nginx
   docker compose rm nginx
   docker compose up -d nginx
   ```

3. Preventive measures:
   - Dodaj resolver z TTL w nginx.conf
   - Użyj Docker DNS name zamiast IP
   - Dodaj health check dla upstream

---

PROMPT 3.3: Diagnostyka Manual Sync Divergence
-----------------------------------------------
Gdy manual sync daje inne wyniki niż scheduled:

1. Porównaj config resolution:
   ```python
   # Worker path
   from app.worker import resolve_feed_config
   # Manual path
   from app.routes.ops_api import resolve_feed_config
   # Czy obie ścieżki dają ten sam wynik?
   ```

2. Sprawdź DB overrides:
   ```sql
   SELECT source_id, key, value FROM app_settings
   WHERE source_id IN ('abusech', 'mwdb', 'misp');
   ```

3. Porównaj env w kontenerach:
   ```bash
   docker inspect ioc-service-worker-1 --format='{{range .Config.Env}}{{.}}\n{{end}}'
   docker inspect ioc-service-app-1 --format='{{range .Config.Env}}{{.}}\n{{end}}'
   ```

4. Fix: Ujednolić logikę w app/services/deps.py
